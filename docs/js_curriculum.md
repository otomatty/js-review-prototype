# JavaScript学習カリキュラム設計書

完全初心者から、Webアプリ実装・アルゴリズム的思考・フレームワーク学習の前提までを段階的に到達させる、長期カリキュラム＋演習システムの設計案。

---

## 1. 全体構造

カリキュラム全体を **5段階のステージ** で構成する。MDN「Dynamic scripting with JavaScript」モジュールの章構成を縦軸、難易度を横軸に取る2軸設計。

```
                  ┌────────────────────────────────────────────────┐
横軸：難易度       │ S0    S1     S2      S3      S4      S5      │
縦軸：MDN章       │ 入門  Progate  Progate  paiza   paiza   応用    │
                  │       前半     後半     初級    中級             │
─────────────────┼────────────────────────────────────────────────│
Ch1 変数          │  ●    ●      ●       ●                       │
Ch2 数値・演算    │       ●      ●       ●       ●               │
Ch3 文字列        │       ●      ●       ●       ●               │
Ch4 配列          │       ●      ●       ●       ●       ●       │
Ch5 条件分岐      │              ●       ●       ●       ●       │
Ch6 ループ        │              ●       ●       ●       ●       │
Ch7 関数          │              ●       ●       ●       ●       │
Ch8 イベント      │                      ●       ●       ●       │
Ch9 オブジェクト  │                      ●       ●       ●       │
Ch10 DOM          │                      ●       ●       ●       │
Ch11 Network/JSON │                              ●       ●       ●│
Ch12 デバッグ     │  ●    ●      ●       ●       ●       ●       │
```

各章は **すべてのステージで段階的に再登場する**(スパイラルカリキュラム)。例：「配列」はS1で「作成・参照」だけ、S2で「push/pop」、S3で「mapで変換」、S4で「ソート＋累積処理」、S5で「2次元配列・行列」と深まる。

### 1.1 ステージ定義

| ステージ | 通称 | 1問の目安時間 | 1問で扱う概念 | 想定モデル |
|---|---|---|---|---|
| **S0 セットアップ** | "はじめての一行" | 3〜5分 | 出力・実行確認のみ | MDN "What is JS / First splash" |
| **S1 文法体験** | Progate前半相当 | 5〜10分 | 1構文・1メソッドに限定 | Progate JavaScript I〜III |
| **S2 文法定着** | Progate後半相当 | 10〜15分 | 1構文 + 既習1〜2個の組合せ | Progate IV〜VII |
| **S3 ロジック入門** | paiza C〜B相当 | 15〜25分 | 標準入力＋単純な処理パターン | paiza Cランク |
| **S4 アルゴリズム** | paiza B〜A相当 | 25〜45分 | 計算量を意識・データ構造選択 | paiza B/Aランク |
| **S5 実装応用** | "動くものを作る" | 45〜90分 | DOM操作・非同期・統合課題 | MDNチャレンジ＋小アプリ |

### 1.2 1ステージ完結ルールではなくスパイラル方式にする理由

要件の「完全初心者レベルから各章を一通り学び、次のステージに進む構成」は1案目として有効ですが、長期カリキュラムでは **スパイラル方式** を推奨します：

- 「変数→演算→文字列→...→デバッグ」と一周してから次のステージに進むのが基本
- ただし **章ごとに必要な深さが違う** ので、たとえばS3の関数の「再帰」は学ばずS4で初出、というように章ごとの登場ステージを上の表で制御する
- 各ステージの最後に **総合チャレンジ問題**(MDNのChallengeに相当)を置き、ステージ卒業条件とする

## 4. レベル適応 補助機能

要件の「ブロックプログラミング/テンプレートの詳細度を複数」は、**スカフォールディングを4段階に切り替えられる** UIで実現する。

### 4.1 4段階のスカフォールド

| Level | 名称 | 説明 | 想定対象 |
|---|---|---|---|
| L4 | **ブロック** | Scratch/Blockly的GUI。コード生成 | S0〜S1の苦手な人 |
| L3 | **穴埋め** | コード骨格に`____`の空欄 | S1〜S2 |
| L2 | **コメント誘導** | `// ここでXXする` のコメントだけ与える | S2〜S3 |
| L1 | **シグネチャのみ** | 関数名・引数・戻り値型のみ | S3〜S4 |
| L0 | **白紙** | 問題文だけ | S4〜S5 |

ユーザーは **問題ごとに難易度を切り替えられる** (例: 同じ問題でもL4で挑戦→クリア→L2で再挑戦)。

### 4.2 ブロックプログラミング(L4)の設計指針

- Blockly / Scratch3 ベースで、対応するJSコードがリアルタイムにエディタに反映される
- ブロックの種類はそのステージで既習の構文に限定(S1の段階でループブロックは出さない)
- 「ブロックで作る → 生成されたJSを見る → 自分で書く」 の3ステップで卒業を促す

### 4.3 穴埋めテンプレート(L3)の例

```js
// 1〜nまでの合計を求める
function sum(n) {
  let total = ____;
  for (let i = ____; i <= ____; i++) {
    total ____;
  }
  return ____;
}
```

### 4.4 自動レベル提案

各問題でユーザーの行動ログを取り、次の問題の初期レベルを自動提案：
- ヒントを2つ以上開いた → 次の問題はL3で開始
- 一発正解＋3分以内 → 次の問題はL1で開始

ユーザーは提案を無視できる。

## 5. 演習システムに必要な機能

### 5.1 必須機能(MVP)

1. **コードエディタ**(Monaco / CodeMirror)
2. **自動採点**: テストケースで`expected_output`照合 + AST的禁止構文チェック(例: S1で`if`使用禁止のときに検出)
3. **段階的ヒント開示**: ヒントをクリックで1つずつ表示、開いた数を記録
4. **解答開示**: 一定時間/ヒント全開後に解答を見られる
5. **進捗マップ**: 章×ステージのグリッドで未着手/進行中/完了を可視化
6. **スカフォールドL0〜L4切替**

### 5.2 学習体験を上げる機能(後回し可)

7. **ブロックエディタ(Blockly)**: L4
8. **AIヒント**: 提出コードを見て次の一言ヒントを生成(LLM API)
9. **エラー解説**: スタックトレースを初心者向けに翻訳
10. **間違い再出題**: 間違えた問題を3日後に再出題(SRS)
11. **コードレビュー機能**: 解いた後に模範解答との差分を表示し、なぜ違うかをLLMで解説
12. **「あなたが書いたコードで何が起きるか」可視化**: ステップ実行・変数の中身トレース

### 5.3 アーキテクチャ(参考)

- フロント: React + Vite + Tailwind + Monaco Editor + Blockly
- バックエンド: Supabase (auth + 進捗保存 + 問題マスタ)
- 採点実行: ブラウザ内サンドボックス(Web Worker + iframe)で安全に実行
- 問題定義: YAMLでGit管理。CIでスキーマバリデーション

→ 既存の社内OSSグループ構成(React+Vite+Tailwind+Supabase)と完全に同じスタックで作れます。

## 8. 設計上の判断ポイント

最後に、ここまでの設計で **意思決定が必要な3点** をまとめます。回答に応じて詳細を詰められます。

1. **スパイラル方式 vs ステージ完結方式**
   本提案はスパイラル(各ステージで全章を一巡)。要件の文面は完結方式に近いので、どちらにするか。

2. **標準入力をどこから入れるか**
   本提案はS3から(paizaライク)。S2の後半から軽く入れる選択肢もあり。

3. **ブラウザ実行 vs Node実行**
   S0〜S2はブラウザで`console.log`、S3〜S4はNode風の標準入力、S5でDOMに戻る、という3段階を提案。1つに統一する案もある(全部ブラウザで擬似標準入力)。
