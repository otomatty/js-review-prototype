# Ch12. デバッグ

`console.log`、 エラーの読み方、 切り分け、 検証。 全ステージに `●` が付いた唯一の章で、 各ステージで「自分の手で原因を切り分ける」 訓練を続ける。

## 含まれるステージ

- **S0** セットアップ — エラーが出たときに「どこで止まったか」 を読む。
- **S1** 文法体験 — `console.log` で値を確かめる癖をつける。
- **S2** 文法定着 — 期待値と実際値の差から原因を切り分ける。
- **S3** ロジック入門 — 関数の入出力を分けて検証する。
- **S4** アルゴリズム — 大きいデータで起きるバグを最小再現に縮める。
- **S5** 設計演習 — 統合課題で起きた不具合を仮説 → 検証で詰める。

## S2 で扱う問題 (8 問)

| # | ID | 主題 |
|---|---|---|
| 01 | `S2-Ch12-01-off-by-one-for` | for ループの境界バグ (`<=` → `<`) |
| 02 | `S2-Ch12-02-condition-flipped` | 比較演算子の向きを直す |
| 03 | `S2-Ch12-03-missing-return` | 関数の return 抜けを直す |
| 04 | `S2-Ch12-04-undefined-trace` | 変数名タイポを直す (totl → total) |
| 05 | `S2-Ch12-05-loose-eq-bug` | `==` を `===` に直す |
| 06 | `S2-Ch12-06-infinite-loop-fix` | while の `i++` 忘れを直す |
| 07 | `S2-Ch12-07-else-if-order` | else if の条件順を直す |
| 08 | `S2-Ch12-08-mutation-bug` | reverse の破壊的変更を防ぐ |

### S2 学習ポイント

- ループ境界 `<` / `<=` の選択。 配列なら `i < arr.length` が定石
- `==` は型変換するので、 数値と文字列が思わぬところで一致する → `===` を使う
- 関数の `return` 抜けは「呼び出し側で `undefined` が出る」 ことで気付く
- 破壊的メソッド (`sort` / `reverse`) は **元配列を変更する**。 元を残したいなら `slice()` でコピーする
- `else if` の条件は **厳しい順** に並べる (大きい数 → 小さい数)

## S4 で扱う問題 (4 問)

| # | ID | 主題 |
|---|---|---|
| 01 | `S4-Ch12-01-fix-binary-search` | 二分探索の境界条件 (`<` → `<=`) を直す |
| 02 | `S4-Ch12-02-fix-float-equal` | 浮動小数の `===` の罠を許容誤差で直す |
| 03 | `S4-Ch12-03-fix-merge-sorted` | マージ後に残った要素を取りこぼすバグを直す |
| 04 | `S4-Ch12-04-fix-counter-factory-capstone` | [卒業課題] 共有された変数による独立性破壊を直す |

### S4 学習ポイント

- **境界条件は最小再現で再現する** — `binarySearch` の末尾要素・1 要素配列のような小さなケースが、 ループ条件 (`<` / `<=`) の選び方を見抜く鍵
- **浮動小数の `===` は危険** — `0.1 + 0.2 !== 0.3`。 `Math.abs(a - b) < epsilon` で許容誤差を持って比較する
- **AND 終了の while は片側に余りが残る** — マージや 2 ポインタの定型では、 終了後に **両方** の残りを処理する 2 本の追加 while がパターン
- **クロージャは「どの変数を捕まえているか」 が全て** — ループ外の変数を捕まえると全関数で共有されてしまう。 ループ内で `let` 宣言するか、 生成関数を分けて独立スコープを作る
- バグ修正の進め方は **「観察 → 仮説 → 最小再現 → 修正」**。 大きいテストケースで気づいたバグも、 小さな入力で再現すると原因を一点に絞れる

## S5 で扱う問題 (3 問)

| # | ID | 主題 |
|---|---|---|
| 01 | `S5-Ch12-01-fix-order-state-machine` | 状態機械の遷移表 / 終端判定 / 遷移関数の 3 関数横断バグを直す |
| 02 | `S5-Ch12-02-fix-cart-total` | カート集計の仕様 vs 実装の差 3 箇所 (重複マージ / 数量 / 割引解釈) を直す |
| 03 | `S5-Ch12-03-fix-merge-intervals-capstone` | [卒業課題] 区間マージの境界バグ 4 箇所 (空 / 未ソート / 隣接 / 包含) を直す |

### S5 学習ポイント

- **状態機械の不整合は遷移表だけ見ても直らない** — 遷移表 (`canTransition`) / 終端判定 (`isFinal`) / 遷移関数 (`transition`) の 3 つが**仕様で一致**して初めて整合する。 ひとつだけ直しても下流の関数が間違った答えを返し続ける
- **仕様書 vs 実装の差は「単体テスト」 では見えない** — 1 個だけ商品を追加するテストは通ってしまうが、 同じ商品を 2 回追加するテストで初めて重複マージのバグが表面化する。 **統合シナリオ**でしか見えないバグを潰すには、 仕様書を読みながら関数の境界を 1 つずつ照合する
- **割引・金額は「percent」 か「flat」 かを必ず仕様で確認** — 同じ `discount` という名前でも、 「10% 引き」 (`* (1 - p/100)`) と「10 円引き」 (`- p`) では結果が桁違いに違う。 名前から推測せず仕様書の数式を読む
- **入力境界は全方位で洗い出す** — `mergeIntervals` のような関数で壊れやすい境界は **空 / 単一 / 隣接 / 包含 / 未ソート / 同一要素**。 「典型ケース」 だけ通って境界ケースで全滅、 は実務で頻発するバグ。 修正は **1 箇所ずつ潰す**: 空配列 → 未ソート → 隣接 → 包含 の順
- **破壊的メソッドは静かに入力を壊す** — `Array.prototype.sort` は元配列を変更する。 `slice()` でコピーしてからソートしないと、 呼び出し元の入力が並び替わる。 「元の入力を破壊しない」 テストは関数設計の鏡
- S5 ステージ全体のテーマは **「複数関数 / 複数概念にまたがるバグを、 最小再現 → 1 箇所修正 → 残るテスト確認 のループで潰す」**
