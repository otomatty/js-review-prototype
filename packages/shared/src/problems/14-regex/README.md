| 03 | `S3-Ch14-03-is-email-shape` | ^ $ アンカーで完全一致 |
| 04 | `S3-Ch14-04-mask-numbers` | replace + g フラグで全置換 |

### S3 学習ポイント

- 正規表現リテラル `/pattern/flags` の構文。 `/\d/` で 1 文字、 `/\d+/` で 1 文字以上の連続
- フラグ `g` 無しは「最初の 1 件」、 `g` 付きは「すべて」
- アンカー `^` `$` で「文字列全体」 にマッチを限定。 単独のパターンは部分一致になる
- `[^@\s]+` のような文字クラスで「除外したい文字」 を指定する

## S4 で扱う問題 (5 問)

| # | ID | 主題 |
|---|---|---|
| 01 | `S4-Ch14-01-parse-date` | キャプチャグループ `(...)` で `YYYY-MM-DD` を分解 |
| 02 | `S4-Ch14-02-is-postal-code` | 量指定子 `{n}` で桁数固定の入力検証 |
| 03 | `S4-Ch14-03-snake-to-camel` | `replace` + コールバック関数で整形 |
| 04 | `S4-Ch14-04-extract-hashtags` | `matchAll` + キャプチャでトークン抽出 |
| 05 | `S4-Ch14-05-parse-query-string-capstone` | [卒業課題] `matchAll` + 2 キャプチャ + `reduce` でオブジェクト化 |

### S4 学習ポイント

- **キャプチャグループ** `(...)` で「マッチした部分文字列」 を後から取り出せる。 `match` の戻り値は
  `[0]` が全体、 `[1]` `[2]` ... がキャプチャに対応する
- **量指定子** `{n}` (ちょうど n 回)、 `{n,m}` (n〜m 回) で **桁数を厳密に制御** する。 入力検証で必須
- `replace(re, callback)` の **コールバック関数** は `(マッチ全体, キャプチャ1, ...)` を受け取り、 戻り値が置換文字列になる。
  整形 (大文字化・小文字化・並び替え) に使う
- `matchAll(re)` は **イテレータ** を返す。 `[...s.matchAll(re)]` で配列化してから `map` / `reduce` に流し込むのが定番

## S5 で扱う問題 (3 問、 卒業課題 1 問)

| # | ID | 主題 |
|---|---|---|
| 01 | `S5-Ch14-01-tokenize-expression` | 名前付きキャプチャ `(?<name>...)` + `matchAll` + `for...of` で 算術式を 3 種類のトークンに分解する |
| 02 | `S5-Ch14-02-parse-log-lines` | `split("\n")` で行に分解 → 各行を `parseLogLine` で名前付きキャプチャパース → null をスキップする 2 段パイプライン |
| 03 | `S5-Ch14-03-parse-frontmatter-capstone` ⭐ | **[S5 卒業課題]** 2 種類の正規表現と名前付きキャプチャを組み合わせ、 Markdown 風文書を `{ meta, body }` に分解する多段パーサー |

### S5 学習ポイント

- **名前付きキャプチャグループ** `(?<name>...)` は、 種別が増えるとすぐ破綻する添字 `m[1]` / `m[2]` を **意味ある名前** に置き換える。 `m.groups.<name>` で取り出し、 マッチしなかった名前は `undefined` になるので 「どのパターンに当たったか」 を 1 つの正規表現で判定できる
- **複数行入力の分解** は **「1 行を見る純粋関数」 と 「全体を集約する関数」 を 2 段** に分けるのが定石。 1 行関数を再利用すれば、 集約側は `split` + `null` フィルタの簡単なロジックに集中できる
- **`[\s\S]` イディオム** は `.` と違って改行にもマッチする 「あらゆる文字」 を表す定番パターン。 `dotAll` フラグを使わなくても複数行をまたぐマッチが書ける
- **非貪欲量指定子 `*?`** を使うのが多段パーサーの要点。 貪欲 `*` だと文書末尾の区切り文字まで吸い込んでしまい、 「フロントマターと本文の境界」 が壊れる
- **マッチ失敗は `null`、 例外は投げない** という設計を貫く。 卒業課題では `ThrowStatement` を AST forbidden にしており、 「失敗を値として持ち回る」 Ch13 S5 の発想をパーサー設計でも継続する
- **アンカー `^`/`$` を必ず付ける** ことで 「行全体」 を厳密に表現する。 アンカー無しの正規表現は 「行のどこかにマッチ」 になり、 入力検証の意図が壊れる
